# 에이전트 확장 및 품질 강화 구현 계획

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 원본 Claude-Code-Game-Studios의 에이전트 역할 8종을 hwire 포맷으로 추가하고, 기존 12개 에이전트에 Safety Protocol / Prohibited Actions / Delegation Structure / Approval Workflow 섹션을 추가하여 품질을 강화한다.

**Architecture:** 기존 agents/ 디렉토리의 마크다운 포맷(한국어, 책임/결정권한/엔진별/협업/hwire통합 구조)을 유지하면서, 원본의 구조적 장점(금지 행동, 보고 체계, 승인 워크플로우, 안전 프로토콜)을 4개 신규 섹션으로 통합한다. 백엔드 DB의 agents 테이블 시드 데이터도 함께 업데이트한다.

**Tech Stack:** Markdown (에이전트 정의), Python/SQLite (백엔드 시드 데이터)

### 추가할 섹션 템플릿

각 에이전트별로 내용을 커스터마이즈하되, 아래 구조를 따른다:

```markdown
## 안전 프로토콜

- **엔진 버전 검증**: 구현 전 `docs/engine-reference/[engine]/VERSION.md`에서 API 호환성 확인. 프로젝트 고정 버전 이후 도입된 API는 플래그 처리
- **ADR 준수**: `docs/architecture/` 디렉토리의 아키텍처 결정 기록(ADR)을 확인하고 구현 가이드라인을 정확히 준수. 불일치 시 리뷰 요청
- **파괴적 변경 금지**: 기존 시스템의 공개 인터페이스를 변경할 때는 반드시 영향 범위를 분석하고 승인 후 진행

## 금지 행동

- [에이전트별 커스텀 금지 항목 3-5개]

## 보고 체계

- **상위 보고**: [보고 대상 에이전트]
- **하위 위임**: [위임 가능 에이전트 또는 "없음"]
- **수평 협력**: [협력 에이전트 목록]
- **에스컬레이션**: 도메인 외 결정이 필요한 경우 [에스컬레이션 대상]에게 상향 보고

## 승인 워크플로우

1. **설계 문서 분석**: 명세 내 지정/미지정 요소를 식별하고 구현 난제를 플래그
2. **아키텍처 질문**: 가정하지 않고 결정 사항을 질문으로 제시
3. **아키텍처 제안**: 구조, 트레이드오프, 근거를 제시하고 사용자 승인 대기
4. **투명한 구현**: 모호성 발견 시 중단하고 명확화 요청
5. **파일 변경 전 승인**: 변경 대상 파일 목록을 제시하고 "진행해도 될까요?" 확인
6. **다음 단계 제안**: 테스트, 코드 리뷰, 리팩터링 옵션 제시
```
